Create `D:/GSDteams/workflows/team-studio.md` as a GSD workflow file (same structure as
`agent-dispatcher.md` and `review-team.md`: `<purpose>`, `<inputs>`, `<step name="...">` blocks).
No YAML frontmatter on the workflow file itself.

The workflow implements these steps:

---

**Step 1: `show_roster` (priority="first")**

Check TEAM.md exists:
```bash
[ -f .planning/TEAM.md ] || echo "MISSING"
```
If missing: inform user "No TEAM.md found at .planning/TEAM.md. Run `/gsd:new-reviewer` to
create your first agent role." Then exit.

Read `.planning/TEAM.md` with the Read tool. Extract from frontmatter: `version`, `roles` list.
For each slug in roles list: find `## Role: {slug}` section, extract the YAML config block
(content between first triple-backtick yaml fence and its closing fence). Parse these fields:
`name`, `focus`, `mode`, `trigger`, `output_type`, `commit`, `enabled`. Apply the same
`enabled` default as the dispatcher: if `enabled` field is absent or null → treat as `true`.
Apply normalizeRole defaults for `mode` (advisory), `trigger` (post-plan), `output_type`
(findings) when version < 2 (numeric comparison).

Display the roster as a markdown table:

```
## Agent Team

**{N} agent(s) configured**

| # | Agent | Mode | Trigger | Output | Status |
|---|-------|------|---------|--------|--------|
| 1 | Security Auditor | advisory | post-plan | findings | active |
| 2 | Doc Writer | autonomous | post-phase | artifact | **DISABLED** |
```

- Enabled agents: Status = `active`
- Disabled agents: Status = `**DISABLED**` (bold for visual distinction)
- Show `#` column for easy reference in action prompts

After the table, show a summary line: "post-plan: N | post-phase: N | pre-plan: N | on-demand: N"

---

**Step 2: `present_actions`**

Use AskUserQuestion to present the action menu:

```
AskUserQuestion([{
  question: "What would you like to do?",
  header: "Team Roster",
  multiSelect: false,
  options: [
    { label: "Enable / Disable", description: "Toggle an agent on or off" },
    { label: "Remove agent",     description: "Delete a role from TEAM.md (with confirmation)" },
    { label: "Run agent now",    description: "Invoke an agent on-demand against a file" },
    { label: "View history",     description: "Show AGENT-REPORT.md entries for an agent" },
    { label: "Add agent",        description: "Create a new agent role" },
    { label: "Done",             description: "Exit the team roster" }
  ]
}])
```

Headers MUST be ≤12 characters (established convention — Phase 05-02 decision).
"Team Roster" is 11 characters — compliant.

Route to the corresponding sub-step based on the user's selection. After each sub-step
completes, return to `present_actions` (loop). Only "Done" exits.

---

**Step 3: `toggle_enabled` (sub-step for "Enable / Disable")**

Re-display the current roster table (abbreviated — agent name and current status only).

AskUserQuestion: "Which agent do you want to toggle?" with one option per agent (label =
agent name, description = current status: "currently active" / "currently disabled") plus
a "Cancel" option.

If Cancel: return to present_actions without changes.

After selection: read current `enabled` value for that role.
- If currently enabled (true or absent): set `enabled: false`
- If currently disabled (false): set `enabled: true` (or remove the field — prefer setting
  `enabled: true` explicitly for clarity)

Write the updated TEAM.md back using the Read-then-Write pattern:
1. Read current TEAM.md
2. Find the `## Role: {slug}` section
3. In its YAML config block: add or update the `enabled:` line
   - If `enabled:` line already exists: replace its value
   - If `enabled:` line does not exist: add it after the last field in the YAML block,
     before the closing triple-backtick fence
4. Write full updated TEAM.md

Announce: "Agent '{name}' {enabled/disabled}. The dispatcher will {now skip / resume} this
agent on the next trigger."

Return to present_actions.

---

**Step 4: `remove_agent` (sub-step for "Remove agent")**

AskUserQuestion: "Which agent do you want to remove?" with one option per agent (label =
agent name, description = mode + trigger summary) plus a "Cancel" option.

If Cancel: return to present_actions.

After selection: read TEAM.md. Extract the full role block for the selected slug.
Show the extracted role block to the user so they can see what will be deleted.

AskUserQuestion confirmation gate:
```
{
  question: "Remove '{role_display_name}' from TEAM.md? This cannot be undone.",
  header: "Remove",
  multiSelect: false,
  options: [
    { label: "Remove", description: "Delete role block and remove from roles list" },
    { label: "Cancel", description: "Return to roster without changes" }
  ]
}
```

If Cancel: return to present_actions without changes.

If Remove:
1. Read current TEAM.md
2. Remove the slug from the frontmatter `roles:` list
3. Remove the role block: find `---` separator + `## Role: {slug}` header through the NEXT
   `---` separator (or EOF). Delete this entire block including both separator lines.
4. Write full updated TEAM.md
5. Announce: "Agent '{role_display_name}' removed from TEAM.md."

Return to present_actions. Re-display the updated roster table before showing the action menu.

---

**Step 5: `invoke_on_demand` (sub-step for "Run agent now")**

AskUserQuestion: "Which agent do you want to run?" with one option per ENABLED agent only
(label = agent name, description = mode + trigger + output_type) plus "Cancel".
(Disabled agents are not shown — cannot invoke a disabled agent on-demand)

If Cancel: return to present_actions.

AskUserQuestion: "What artifact should this agent run against?" Present options:
- List the 3 most recently modified files in `.planning/phases/` (find via Bash:
  `find .planning/phases -name "*.md" -not -name "AGENT-REPORT.md" | xargs ls -t 2>/dev/null | head -5`)
- Plus an "Enter path manually" option

If "Enter path manually": prompt user for the path (ask as a follow-up question).

Determine `phase_id` for AGENT-REPORT.md path:
- If the artifact path contains `.planning/phases/{slug}/`: extract `{slug}` as the phase_id
- Otherwise: use the most-recently-modified phase directory:
  `ls -td .planning/phases/*/ | head -1 | xargs basename`

Read the artifact file content with the Read tool.
Read the agent's full role block from TEAM.md (the `## Role: {slug}` section).

Spawn a Task() for the agent:
- For advisory agents: spawn with role_definition + artifact content as prompt context.
  Prompt format: "You are acting as the following agent role:\n\n{role_block}\n\nReview the
  following artifact and produce your output:\n\n{artifact_content}"
- For autonomous agents: spawn with role_definition + artifact path + trigger context "on-demand".

Display the Task() result inline in the conversation.

Log to AGENT-REPORT.md:
1. Check if `.planning/phases/{phase_id}/AGENT-REPORT.md` exists
2. If it does NOT exist: create it with a header:
   `# Agent Report — Phase {phase_id}\n\n---\n`
3. Append this section:
   ```
   ## On-Demand: {agent-slug} — {ISO timestamp}

   **Agent:** {agent display name}
   **Mode:** {advisory | autonomous}
   **Artifact:** {artifact path}
   **Invoked by:** on-demand (/gsd:team)

   ### Result

   {agent output}

   ---
   ```
4. Write the updated file (Read-then-Write for existing; Write-new for new file)

Announce: "Result logged to `.planning/phases/{phase_id}/AGENT-REPORT.md`"

Return to present_actions.

---

**Step 6: `view_history` (sub-step for "View history")**

AskUserQuestion: "Which agent's history do you want to view?" with one option per agent
plus an "All agents" option and a "Cancel" option.

If Cancel: return to present_actions.

Search for AGENT-REPORT.md files:
```bash
find .planning/phases -name "AGENT-REPORT.md" 2>/dev/null
```

For each found file:
- Read its content
- If "All agents": display all entries from all AGENT-REPORT.md files
- If specific agent: filter to only `## On-Demand: {agent-slug}` sections matching the slug

If no AGENT-REPORT.md files found: announce "No agent history found. Run an agent on-demand
to create history."

Display the matching entries inline.

Return to present_actions.

---

**Step 7: `add_agent` (sub-step for "Add agent")**

Inform the user:
"To add an agent:
- For a **review-team role** (advisory, post-plan reviewer): run `/gsd:new-reviewer`
- For a full **agent role** (any mode, trigger, or output type): `/gsd:new-agent` is available
  in a future update (Phase 11)

Run `/gsd:new-reviewer` now to create a reviewer role."

Return to present_actions.
